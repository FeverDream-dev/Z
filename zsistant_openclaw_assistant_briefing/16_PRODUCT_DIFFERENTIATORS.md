# Zsistant Differentiators

Zsistant should not only copy existing products. It should have its own identity.

## Differentiator 1: Assistant manager first

Most chat products organize by conversations.

Zsistant should organize by assistants.

The user manages digital workers, not only threads.

## Differentiator 2: Multi-channel by design

Assistants should live across Web UI, CLI, Telegram, WhatsApp, Slack, Discord, and future surfaces.

The Web UI is the control center, not the only place the assistant exists.

## Differentiator 3: Chrome/browser-native workflows

Browser control should be central.

A Zsistant assistant should be able to inspect websites, capture screenshots, test UI, and use browser sessions when permitted.

## Differentiator 4: Chronological jobs

Scheduled jobs and timelines should be first-class.

The user can see what the assistant will do, what it did, and what failed.

## Differentiator 5: Developer Mode for agents

Developer Mode should reveal tool traces, provider routing, MCP calls, model events, browser actions, and raw diagnostics.

This makes Zsistant more trustworthy and useful for builders.

## Differentiator 6: Honest capability states

Instead of pretending everything works, Zsistant should clearly show:

- Not built
- Needs setup
- Connected
- Working
- Failed
- Needs approval

Honesty is a product advantage.

## Differentiator 7: Assistant operating desk

Each assistant should have a desk:

- Chat
- Jobs
- Channels
- Tools
- Knowledge
- Memory
- Logs
- Browser
- Settings

This makes assistants understandable and manageable.

## Differentiator 8: Multi-agent delegation

Assistants can route tasks to other assistants.

Examples:

- Personal Assistant delegates UI inspection to Browser QA Assistant.
- Browser QA Assistant delegates code fix planning to Coding Supervisor.
- Research Assistant sends durable facts to Memory Curator.

## Differentiator 9: Local/control-friendly philosophy

Where possible, Zsistant should favor user control:

- User-owned configuration
- Clear provider credentials
- Inspectable logs
- Explicit permissions
- Exportable data
- Avoid hidden black boxes

## Differentiator 10: Beautiful normal mode, powerful developer mode

Normal users should not drown in technical details.

Power users should not be locked out of observability.

Zsistant should serve both by separating Normal Mode and Developer Mode.
